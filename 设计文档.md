## 1. 架构概述

本项目采用**模块化单体架构 (Modular Monolith)**。系统在物理代码层面划分为多个高内聚、低耦合的 Maven/Gradle 模块，但在部署和运行时作为一个统一的 Spring Boot 进程启动。

- **核心优势**：兼顾微服务的清晰边界与单体应用的高效部署；未来可根据业务规模，将任意业务模块无缝拆分为独立的微服务。
- **核心技术栈**：Spring Boot 4.x + Spring Security + MyBatis-Plus + PostgreSQL + Redis + flyway 。
- JDK 25

## 2. 工程模块划分与职责流向

项目基于父子工程构建，自下而上依赖，严禁循环依赖和越级依赖。

Plaintext

```
OmniGate_backend (Root Project - 仅用于统一版本管理)
 │
 ├── omnigate-common (基础设施层 - 最底层)
 │    ├── 职责：提供全局共享的配置、工具、异常处理和安全基座。
 │    └── 特性：不包含任何具体业务逻辑。
 │
 ├── omnigate-user (系统用户模块 - 核心业务层)
 │    ├── 依赖：omnigate-common
 │    ├── 职责：系统管理员/使用者的注册、登录、RBAC 权限校验。
 │    └── 特性：负责双 Token 的签发与续期。
 │
 ├── omnigate-google (谷歌账号模块 - 业务层)
 │    ├── 依赖：omnigate-common
 │    └── 职责：管理 Google 账号资产池，对外暴露独立的 `/api/google/*` 接口。
 │
 ├── omnigate-chatgpt (ChatGPT账号模块 - 业务层)
 │    ├── 依赖：omnigate-common
 │    └── 职责：管理 ChatGPT 账号资产池，对外暴露独立的 `/api/chatgpt/*` 接口。
 │
 ├── omnigate-github (GitHub账号模块 - 业务层)
 │    ├── 依赖：omnigate-common
 │    └── 职责：管理 GitHub 账号资产池，对外暴露独立的 `/api/github/*` 接口。
 │
 └── omnigate-bootstrap (启动与组装层 - 最顶层)
      ├── 依赖：omnigate-user, omnigate-google, omnigate-chatgpt, omnigate-github
      ├── 职责：系统的总入口。
      └── 特性：仅包含 `Application.java` 启动类和全局 `application.yml` 配置文件，无业务代码。
```

## 3. 核心依赖清单 (Dependencies)

### 3.1 基础设施与持久层 (Common Module 引入)

- **Web 核心**: `spring-boot-starter-web` (RESTful API 基础)
- **持久层框架**: `mybatis-plus-boot-starter` (简化 CRUD)
- **数据库驱动**: `postgresql` (PostgreSQL 官方驱动)
- **参数校验**: `spring-boot-starter-validation` (接口入参校验)
- **工具箱**: `hutool-all` (涵盖 AES 加解密、字符串处理等核心工具)
- **样板代码省略**: `lombok`
- **接口文档**: `springdoc-openapi-starter-webmvc-ui` (Swagger 3 规范)

### 3.2 安全与缓存机制 (Common & User Module 按需引入)

- **安全框架**: `spring-boot-starter-security` (核心鉴权)
- **JWT 组件**: `jjwt-api`, `jjwt-impl`, `jjwt-jackson` (Token 生成与解析)
- **缓存中间件**: `spring-boot-starter-data-redis` (托管 Refresh Token)
- **Redis 连接池**: `commons-pool2` (Lettuce 连接池必须依赖)
- **JSON 扩展**: `jackson-datatype-jsr310` (处理 Java 8 日期时间的 Redis 序列化)

## 4. 数据库与数据安全规范

1. **完全物理隔离**：不同的业务模块对应不同的数据库表（如 `sys_user`, `acc_google`, `acc_chatgpt` 等），模块间禁止在 SQL 层面进行 `JOIN` 查询。
2. **灵活扩展性**：充分利用 PostgreSQL 的 `JSONB` 数据类型，设立 `ext_info` 字段，用于存储不同平台特有的属性（如 GitHub PAT, ChatGPT Session Key 等）。
3. **核心资产加密**：第三方平台的密码、备用验证码等敏感信息，**绝对禁止明文落库**。
   - **实现方案**：在 `omnigate-common` 中封装基于 Hutool AES 的 MyBatis-Plus `TypeHandler`，实现数据在持久化前自动加密，在查询结果映射时自动解密。

## 5. 认证鉴权架构 (双 Token 机制)

系统采用 **“严格无状态的 Access Token + Redis 托管 Refresh Token”** 的方案，兼顾极致性能与安全管控。

1. **登录签发**：用户登录成功后，下发短效 `AccessToken` (例如 30 分钟) 和长效 `RefreshToken` (例如 7 天)。仅将 `RefreshToken` 存入 Redis。
2. **业务鉴权**：系统内的 API 拦截器仅对 `AccessToken` 进行 JJWT 的内存验签，**不查询数据库和 Redis**，保障核心业务接口的高吞吐量。
3. **无感刷新**：`AccessToken` 过期后，前端使用 `RefreshToken` 调用刷新接口。后端校验 Redis 中的 `RefreshToken` 状态，有效则下发新 Token；若 Redis 中无记录（管理员踢出或过期），则要求重新登录。

## 6. API 路由规范

采用**完全独立的 RESTful API** 风格，各模块各自维护其 Controller。

- **用户模块**：`/api/auth/login`, `/api/auth/refresh`, `/api/users/*`
- **谷歌模块**：`/api/google/accounts/*`
- **ChatGPT 模块**：`/api/chatgpt/accounts/*`
- **GitHub 模块**：`/api/github/accounts/*`
